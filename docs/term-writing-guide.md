# 用語執筆ガイド（初級/中級/上級）

このガイドは、`src/content/terms/*.md` の本文品質をそろえるための執筆基準です。

## 1. 読者レベル定義

- 初級: 略語を初めて見た人
- 中級: 定義はわかるが、実務での使い方が曖昧な人
- 上級: 設計・導入・運用の判断が必要な人

## 2. レベルごとの書き方

### 初級（短く、結論先出し）

書く順番:

1. 一言（20字前後）
2. 軽い説明（1〜2段落）
3. 任意で比喩（1段落）

ルール:

- 最初の1文で意味がわかるようにする
- つまずき論点や例外運用は書かない
- 専門語で専門語を説明しない
- 「結論が後ろ」にならないようにする
- カタカナ語は、より平易な日本語に言い換えられるなら言い換える（例: スイート -> サービス一式）

### 中級（説明できる状態にする）

書く順番:

1. 何をする仕組みか
2. どう動くか（2〜4 ステップ）
3. どこで使うか（典型シーン）
4. 何と混同しやすいか（隣接概念との違い）

ルール:

- 2〜5段落で収める
- 実務の会話で出る単語を優先する
- 仕様の細部は上級へ回す
- 専門語を使う場合は、同じ文で短い言い換えを添える
- 読後に「自分の言葉で説明できる」構造にする
- 仕組みの流れを 2〜4 ステップで必ず書く

### 上級（判断材料を明示）

書く順番:

1. アーキ設計（構成・境界・連携）
2. 運用設計（監視・権限・変更管理・監査）
3. 障害/変更時の論点（詰まりどころ・復旧）
4. これで実際にできること（3〜5項目）

ルール:

- 長くてもよいが、判断軸を先に示す
- 理想論だけで終わらせず、失敗しやすい点を入れる
- 略語や製品名を並べるだけで終わらせず、「何を決める話か」を日本語で先に書く
- 「どの選択肢を選ぶと、何が楽になり、何が重くなるか」を明示する
- 「導入後に何が改善できるか」を、業務目線の動詞（削減/防止/自動化/可視化）で書く

## 3. コピペ用ひな形

```md
## 初級

（一言：20字前後）

（軽い説明：1〜2段落）

（任意）補足（比喩）：

## 中級

（どこで使うか：典型シーン）

（何とセットか：隣接概念・対比）

（どう動くか：2〜4ステップ）

（混同しやすい用語との違い）

## 上級

### アーキ設計

（構成・境界・連携）

### 運用設計

（監視・権限・変更管理・監査）

### 障害/変更時の論点

（詰まりどころ・復旧）

### これで実際にできること

- （できること1）
- （できること2）
- （できること3）
```

## 4. 公開前チェック

初級:

- 1文目だけ読んで意味がわかるか
- 難語の連鎖が起きていないか

中級:

- 「何をする仕組みか」を1文で言えるか
- 流れを2〜4ステップで説明できるか
- 隣接概念との違いが1つ以上あるか

上級:

- アーキ設計と運用設計の両方が書かれているか
- 障害/変更時の詰まりどころが具体化されているか
- 「実際にできること」が3項目以上あるか

## 5. 運用ルール（推奨）

- 執筆順は必ず「初級 -> 中級 -> 上級」
- 1回の作業単位は5〜8語
- バッチ単位でレビューし、都度全件見直しはしない
