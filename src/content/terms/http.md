## 初級

ブラウザとサーバーのあいだで「ページやデータをどう取りに行くか」の約束です。

補足（比喩）：郵便の「宛名・種類・返信ルール」に近いイメージです。

## 中級

やり取りは **リクエスト（依頼）** と **レスポンス（応答）** のペアで進みます。URL は「どこを」「どのメソッド（GET / POST など）で」触るかを表し、ヘッダーは認証情報や送る形式（JSON など）、ステータスコード（200 / 404 / 500 など）は結果の種類を短く伝えます。HTTPS は、この HTTP の通信路に暗号化を足したもので、中身の「依頼と応答の形」は HTTP の延長として理解できます。

実務では「キャッシュで古い表示が残る」「リダイレクトで URL が変わる」「CORS でブラウザがブロックする」など、HTTP の外側の設定とセットでトラブルが出ることが多いです。

## 上級

HTTP/1.1 のキープアライブ、HTTP/2 の多重化、HTTP/3（QUIC）のような **伝送効率** の進化が論点になります。キャッシュは `Cache-Control` / `ETag` / `Vary` の組み合わせで挙動が変わり、API では `Authorization` や `Content-Type`、レート制限ヘッダーと組み合わせて設計します。

プロキシ・CDN・TLS 終端の手前でヘッダが書き換わること、HTTP/2 のサーバープッシュが推奨されにくくなっていること、セキュリティ上 `Set-Cookie` の属性（`Secure` / `HttpOnly` / `SameSite`）をどう付けるかも、運用レベルでは必須の論点です。
