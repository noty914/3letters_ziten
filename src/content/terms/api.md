## 初級

アプリが別のシステムに「データをください」「処理をお願いします」と頼むときの窓口です。

補足（比喩）：レストランの「メニュー」と「厨房へのオーダー方法」に近いです。

## 中級

多くは **HTTP 上の URL** でエンドポイントが表され、JSON やクエリで入力を渡し、JSON で結果を受け取ります。公開 API を提供する側はバージョンや利用規約を明示し、呼び出す側はエラー時のリトライやタイムアウト、認証トークンの更新を組み込みます。

「REST っぽい URL 設計」「GraphQL の 1 エンドポイント」「社内だけの非公開 API」など、見え方は違っても **リクエストとレスポンスの契約** が共通の骨格です。

## 上級

設計の中心は **スキーマ（型）・後方互換・廃止予告（deprecation）・認可モデル** です。OAuth2 / OIDC と組み合わせた認可、スコープとトークンの寿命、クライアントごとのレート制限、gRPC のようなバイナリ RPC との使い分け、Webhook（サーバからクライアント側 URL へのコールバック）など、スタイルごとにトレードオフがあります。

公開 API では破壊的変更の出し方（メジャーバージョン、ヘッダでの sunset）、ドキュメント生成（OpenAPI）、サンドボックス環境の提供が、利用者体験とセキュリティの両面で効いてきます。
